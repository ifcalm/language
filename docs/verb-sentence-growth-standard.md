# 动词句子生长数据生成与校验规范

> 版本：1.0  
> 状态：正式采用  
> 适用范围：`verbs`、`verb_paths` 以及 `verb_paths.growth_json`

句子结构 SVG 的节点拆分、箭头方向、嵌套动作和视觉验收规则，以以下文件为唯一权威标准：

- [`sentence-svg-diagram-standard.md`](./sentence-svg-diagram-standard.md)

本文件中如有任何历史描述与该规范冲突，以句子结构 SVG 绘制规范为准，冲突规则视为废弃。

## 1. 目的

本规范用于指导 English Orbit 动词学习数据的生成、校对和验收。

目标不是向学习者灌输语法术语，而是通过句子逐步生长和树状动画，让学习者形成以下直觉：

1. 先找到句子中的核心动作。
2. 再看围绕动作形成的主干。
3. 最后理解时间、条件、目的、方式、程度等信息如何补充句子。

本规范同时用于控制批量数据质量，避免剩余动词出现模板化、语义生硬、关系错误或 JSON 无法展示等问题。

## 2. 数据职责

### 2.1 `verbs`

`verbs` 是动词基础主表，只保存稳定的基础信息：

- 动词或动词短语
- 规范化查询文本
- 中文核心义
- 是否为动词短语

### 2.2 `verb_paths`

`verb_paths` 保存某个动词在一个具体含义和场景下的学习路径。

一个动词可以有多条路径，但第一阶段不要求所有动词都具备多个义项。只有当不同含义或使用方式确实值得独立学习时，才新增路径。

### 2.3 `growth_json`

`growth_json` 是句子生长动画的权威数据，负责保存：

- 句子词块节点
- 节点之间的关系
- 逐步展示顺序
- 每一步的英文、中文和学习提示

页面的句子生长卡片与树状动画均以 `growth_json` 为主要数据来源。

### 2.4 `steps_json`

`steps_json` 是旧版兼容字段。

在该字段正式删除前：

- 必须保持合法 JSON。
- 句子内容必须与 `growth_json.steps` 语义一致。
- 新数据应优先从 `growth_json` 派生 `steps_json`，避免分别编写后产生偏差。

## 3. 核心学习原则

### 3.1 不使用复杂语法术语

学习者可见内容只使用：

- 动词
- 主干
- 修饰
- 时间
- 条件
- 场景
- 方式
- 程度
- 原因
- 目的
- 结果
- 范围
- 对象细节

学习者页面不得出现下列教学表达：

- 及物动词、不及物动词
- 主语、宾语、补语、定语、状语
- SVO、SVOC 等句型缩写
- 需要预先理解语法体系的专业术语

数据整理者可以在内部判断动词的搭配能力和句子结构，但不能将这些术语写入学习内容。

### 3.2 不统一套用固定模板

句子步骤数量必须由动词本身和当前场景决定。

- 简单路径可以较短。
- 信息关系丰富的路径可以更长。
- 不要求所有动词都使用相同数量的步骤。
- 不要求所有动词都按“时间 → 条件 → 目的”的顺序扩展。

通常建议包含 2 至 5 个数据库步骤：

1. 第一步必须是主干句。
2. 后续每一步增加一个清晰的信息单位。
3. 超过 5 步时，应确认每一步都具有独立学习价值。

前端会在数据库步骤之前自动增加“只看动词”的播放阶段，因此 `growth_json.steps` 不需要额外保存该阶段。

### 3.3 每一步只解决一个问题

每一步应当让学习者能够回答一个更具体的问题，例如：

- 谁或什么在执行动作？
- 动作涉及什么？
- 动作发生在什么时候？
- 动作发生在什么情况下？
- 动作在哪里发生？
- 动作以什么方式完成？
- 动作为什么发生？
- 动作带来什么结果？

同一步不应一次引入多个互不相关的大块信息。

### 3.4 信息只能逐步增加

从主干句到最终句的过程应保持连续：

- 已出现的核心含义不能在下一步消失。
- 不应无理由替换主干中的人物、对象或动作。
- 新句必须能够从上一步自然扩展而来。
- 为保证表达自然，可以进行必要的冠词、单复数或指代调整，但不得改变原意。

优先保持主干措辞稳定，避免学习者在观察新增内容时还要重新识别整个句子。

## 4. 场景选择标准

### 4.1 场景优先级

场景按以下顺序选择：

1. 动词在技术语境中自然、常见时，优先使用技术场景。
2. 职场场景更能体现常见用法时，使用职场场景。
3. 日常场景明显更自然时，使用日常场景。

“技术场景优先”不等于强制技术化。不得为了符合产品定位而制造不自然的技术句子。

### 4.2 技术场景要求

技术句子应来自开发者真实可能遇到的表达，例如：

- 文档和 API
- 代码与配置
- 构建、测试和部署
- 日志、错误和调试
- 缓存、队列和数据库
- 安全、权限和输入校验

避免只替换几个技术名词的伪技术模板。

### 4.3 `scene` 字段

优先复用现有稳定场景值，例如：

- `api`
- `ci`
- `cli`
- `code`
- `config`
- `debug`
- `deploy`
- `frontend`
- `queue`
- `release`
- `runtime`
- `security`
- `system`

新增场景值前，应确认前端是否需要补充对应中文显示名称。

## 5. 英文句子质量标准

### 5.1 基本要求

每个英文句子必须：

- 语法正确。
- 含义清晰。
- 搭配自然。
- 符合现代英语常见表达。
- 在上下文中能够独立理解。
- 使用一致的时态和指代。
- 正确使用冠词、单复数、介词和标点。

### 5.2 动词使用准确

当前路径中的动词必须与 `meaning_zh` 表示同一个具体含义。

需要内部确认：

- 动词能否自然连接当前人物或事物。
- 动词能否自然连接当前对象或后续词块。
- 介词或副词是否属于该动词的常见搭配。
- 短语动词是否保持完整，不能拆掉决定含义的部分。
- 动词形式是否与句子中的人称、数量和时态一致。

这些判断只用于内容质量控制，不在页面中展示语法名称。

### 5.3 长度要求

长度不作为机械硬指标，但应满足阅读训练需要：

- 主干句应尽量短，通常只表达一个核心动作。
- 中间句应能明显看出新增信息。
- 最终句应自然完整，但不追求复杂和冗长。
- 单个最终句通常控制在一次阅读能够理解的范围内。

如果最终句需要多次回看才能找到主干，应拆分路径或减少修饰内容。

### 5.4 禁止内容

不得生成：

- 语法正确但现实中几乎不会使用的句子。
- 只为增加长度而堆叠的修饰语。
- 逻辑自相矛盾的条件、目的或结果。
- 指代不清的 `it`、`they`、`this`、`that`。
- 对开发流程、技术概念或日常常识存在明显误解的表达。
- 直接复制词典、教材、文章或其他受版权保护内容的例句。

所有句子应由项目独立编写。

## 6. 中文内容质量标准

### 6.1 中文核心义

`meaning_zh` 只表达当前路径中实际使用的含义，不堆叠该动词的全部词义。

### 6.2 逐步翻译

每一步 `sentence_zh` 必须：

- 与对应英文信息量一致。
- 准确体现本步新增内容。
- 不提前翻译后续尚未出现的信息。
- 使用自然中文，而不是机械逐词翻译。
- 保持人物、对象、时间和逻辑关系一致。

### 6.3 学习提示

`note_zh` 应当：

- 简短说明本步新增加了什么。
- 使用学习者能直接理解的语言。
- 避免重复整句翻译。
- 避免复杂语法解释。

推荐形式：

- “这一步说明动作发生在什么时候。”
- “这一步让处理范围更具体。”
- “这一步说明为什么要执行这个动作。”

不推荐形式：

- “这里添加了目的状语。”
- “该动词后接宾语并构成 SVO 结构。”
- “本句属于某某从句。”

## 7. `verb_paths` 字段校验

| 字段 | 必须满足 |
|---|---|
| `id` | 全表唯一、稳定、使用小写短横线格式，能够反映动词和路径。 |
| `verb_id` | 必须存在于 `verbs.id`。 |
| `verb` | 必须与对应 `verbs.verb` 一致。 |
| `title` | 简短描述当前路径，如 `cache the result`，不能使用空泛标题。 |
| `meaning_zh` | 与当前路径中的实际动词含义一致。 |
| `core_sentence_en` | 必须等于 `growth_json.steps` 第一步的 `sentence_en`。 |
| `core_sentence_zh` | 必须等于 `growth_json.steps` 第一步的 `sentence_zh`。 |
| `full_sentence_en` | 必须等于 `growth_json.steps` 最后一步的 `sentence_en`。 |
| `full_sentence_zh` | 必须等于 `growth_json.steps` 最后一步的 `sentence_zh`。 |
| `scene` | 使用稳定、可理解的场景标识。 |
| `steps_json` | 合法 JSON，并与 `growth_json.steps` 保持语义一致。 |
| `growth_json` | 合法 JSON，并通过本规范的结构、关系和语言校验。 |

## 8. `growth_json` 结构规范

### 8.1 顶层结构

```json
{
  "nodes": [],
  "links": [],
  "steps": []
}
```

三个字段均为必填数组，不得为 `null`。

### 8.2 `nodes`

每个节点表示一个能够独立观察的词块。

```json
{
  "id": "the-service",
  "text": "the service",
  "kind": "core"
}
```

校验要求：

- `id` 在当前路径内唯一。
- `text` 非空，并且能在至少一个步骤句子中找到对应表达。
- `kind` 只能是 `action`、`core` 或 `modifier`。
- 每条路径必须至少有一个 `action` 节点。
- 每条路径必须能够识别一个主要动作；存在嵌套事件时，应增加对应的 `action` 节点，不得把独立动作降级为普通修饰文本。
- 所有 `action.text` 都必须是当前句子中实际使用的动词或动词短语形式。
- 主干中的其他稳定词块使用 `core`。
- 后续增加的说明性词块使用 `modifier`。
- 一个节点应保持为一个有意义的词块，不能把无关内容塞入同一节点。

### 8.3 `links`

每条连线表示节点之间的结构关系。

```json
{
  "id": "for-a-short-time->caches",
  "from": "for-a-short-time",
  "to": "caches",
  "kind": "modifier",
  "label": "说明缓存时间"
}
```

校验要求：

- `id` 在当前路径内唯一。
- `from` 和 `to` 必须引用已存在节点。
- 不允许节点连接到自己。
- 不允许重复表达同一关系的连线。
- `kind` 只能是 `core` 或 `modifier`。
- `label` 必须准确说明关系，不能使用无意义的统一文案。

关系方向：

- 主干关系使用 `action -> core`。
- 主干连线只表示核心结构关联，不表示某一方“修饰”另一方。
- 修饰关系使用 `modifier -> 被说明节点`。
- 修饰整个动作或事件时，指向 `action`。
- 修饰某个具体主干词块时，指向对应 `core` 节点。
- 修饰另一个修饰词块时，可以指向对应 `modifier` 节点。

不得把所有修饰内容无判断地连接到动词节点。

### 8.4 `steps`

```json
{
  "step_no": 2,
  "label": "加一点时间",
  "sentence_en": "The service caches the result for a short time.",
  "sentence_zh": "服务会短时间缓存这个结果。",
  "show_nodes": ["for-a-short-time"],
  "show_links": ["for-a-short-time->caches"],
  "focus_node": "for-a-short-time",
  "note_zh": "这一步说明缓存保留多久。"
}
```

校验要求：

- `step_no` 从 1 开始连续递增，不得重复或跳号。
- 第一步 `label` 必须为“主干”。
- 后续 `label` 根据实际新增信息编写，不固定套用。
- `sentence_en` 和 `sentence_zh` 均不能为空。
- `show_nodes` 只列出本步新出现的节点。
- `show_links` 只列出本步新出现的连线。
- `focus_node` 通常应为本步新出现的重点节点。
- 第一步应展示 `action` 和全部主干节点及其主干连线。
- 最后一步结束后，所有需要展示的节点和连线都必须已经出现。
- 每一步新增节点必须能在该步 `sentence_en` 中找到对应表达。
- 每一步新出现的连线两端，在该步结束时都必须已经可见。

## 9. 句子与树的对应规则

### 9.1 主干

主干应表达句子最核心、最稳定的事件。

例如：

```text
The service caches the result.
```

对应节点：

- `the service`：`core`
- `caches`：`action`
- `the result`：`core`

对应主干连线：

- `caches -> the service`
- `caches -> the result`

### 9.2 修饰

例如新增：

```text
for a short time
```

它说明缓存持续多久，因此应连接到 `caches`，而不是连接到 `the service` 或 `the result`。

再例如：

```text
with a clear message
```

如果它描述的是 `an error` 的具体内容，应连接到 `an error`，而不是无判断地连接到动作节点。

### 9.3 页面视觉含义

- 虚线框用于标识主干整体。
- 核心关系使用黑色实线箭头，从动作指向它直接带出的核心部分。
- 修饰关系使用绿色实线箭头，从修饰内容指向真正被说明的节点。
- 动词节点始终是句子结构的视觉核心。
- 逐步动画只控制节点和连线的出现顺序，不得省略最终关系，不得改变箭头方向。
- 页面空间不足时应增加高度或调整布局，不得通过删除箭头、合并动作或隐藏重要节点简化结构。
- 完整规则以 [`sentence-svg-diagram-standard.md`](./sentence-svg-diagram-standard.md) 为准。

## 10. 结构自动校验清单

以下问题应当通过程序自动发现，并阻止数据进入正式数据库：

- `growth_json` 不是合法 JSON。
- `nodes`、`links` 或 `steps` 缺失或类型错误。
- 没有 `action` 节点。
- 存在多个动作时，未能识别主要动作与嵌套动作之间的关系。
- 具有独立事件意义的嵌套动作被错误保存为普通 `modifier` 节点。
- 节点 ID、连线 ID 或步骤编号重复。
- 连线引用不存在的节点。
- `show_nodes` 引用不存在的节点。
- `show_links` 引用不存在的连线。
- `focus_node` 引用不存在的节点。
- 步骤编号不连续。
- 第一步不是“主干”。
- 第一步未展示动作节点。
- 某个节点或连线从未被任何步骤展示。
- 后续步骤删除了前一步已经表达的核心信息。
- `core_sentence_en/zh` 与第一步不一致。
- `full_sentence_en/zh` 与最后一步不一致。
- `verb_id` 不存在或 `verb` 与主表不一致。
- 英文或中文句子为空。

## 11. 人工语言校验清单

每条路径必须逐项确认：

### 英文

- [ ] 动词含义与 `meaning_zh` 一致。
- [ ] 动词形式正确。
- [ ] 搭配、介词和词序自然。
- [ ] 每一步都是上一步的自然扩展。
- [ ] 最终句符合真实使用习惯。
- [ ] 没有明显的中式英语。
- [ ] 指代对象清楚。
- [ ] 技术表达没有事实性错误。
- [ ] 句子为项目独立编写。

### 中文

- [ ] 中文翻译自然准确。
- [ ] 中英文信息量一致。
- [ ] 每一步新增信息均被正确翻译。
- [ ] `note_zh` 简短且适合学习者。
- [ ] 未出现复杂语法术语。

### 学习过程

- [ ] 主干足够短且容易识别。
- [ ] 步骤数量由当前动词决定，而非套用统一模板。
- [ ] 每一步只增加一个清晰信息单位。
- [ ] 步骤标签准确描述新增信息。
- [ ] 最终句不过度复杂。

### 树状关系

- [ ] 动词是唯一动作节点。
- [ ] 主干节点与动作节点关系准确。
- [ ] 每个修饰节点都连接到真正被说明的节点。
- [ ] 修饰节点没有全部机械连接到动词。
- [ ] 箭头方向为“修饰内容 → 被说明内容”。
- [ ] 动画逐步播放时不存在孤立节点或无效连线。

## 12. 错误等级

### 阻断错误

出现以下问题时不得录入：

- JSON 无效。
- 句子语法错误。
- 动词含义使用错误。
- 树状关系方向错误。
- 节点或连线引用不存在。
- 主干句、完整句与步骤数据不一致。
- 中文翻译改变原意。

### 重要错误

应在批次提交前修正：

- 场景生硬。
- 步骤增长不连续。
- 修饰关系过于粗糙。
- 指代不清。
- 最终句过长或信息堆叠。
- 标签和实际新增内容不一致。

### 一般问题

可以在校对阶段调整：

- 中文措辞不够顺畅。
- `note_zh` 略显重复。
- 节点 ID 命名不够清晰。
- 标点或大小写风格不统一。

## 13. 推荐生产流程

每一批动词按照以下顺序处理：

1. 确认动词在当前路径中的具体含义。
2. 选择最自然的技术、职场或日常场景。
3. 编写简短稳定的主干句。
4. 根据动词实际用法决定扩展步骤。
5. 编写每一步英文和中文。
6. 划分 `action`、`core` 和 `modifier` 节点。
7. 明确每个修饰词块真正说明的节点。
8. 生成 `growth_json`。
9. 从 `growth_json` 派生 `verb_paths` 摘要字段和兼容 `steps_json`。
10. 运行结构自动校验。
11. 完成人工语言与树状关系校验。
12. 在页面实际播放一次，确认动画顺序和视觉关系。

建议按可复核的小批次生成数据，而不是一次生成全部剩余动词后再集中返工。

## 14. 完成定义

一个动词路径只有同时满足以下条件，才视为完成：

1. `verb_paths` 所有必填字段完整。
2. `growth_json` 结构合法。
3. 英文和中文通过人工校验。
4. 主干到完整句的增长过程自然。
5. 树状关系准确表达“哪部分说明哪部分”。
6. 页面能够从动词阶段自动播放到最终步骤。
7. 句子卡片中每一步新增内容能够正确高亮。
8. 动画中不存在节点重叠、孤立节点或错误箭头。
9. `core_sentence_*`、`full_sentence_*` 与步骤数据一致。
10. 数据不包含版权来源不明的复制例句。

只有满足上述完成定义，才计入“已补录动词数据”。

## 15. 自动校验脚本

项目提供：

```text
scripts/check-verb-paths.mjs
```

默认校验本地 D1：

```bash
npm run verbs:validate
```

严格校验远端 D1：

```bash
npm run verbs:validate:remote
```

其他用法：

```bash
# 只检查某个动词、动词 ID 或路径 ID
node scripts/check-verb-paths.mjs --remote --verb=cache

# 输出便于程序处理的 JSON
node scripts/check-verb-paths.mjs --remote --json

# 将启发式警告也视为失败
node scripts/check-verb-paths.mjs --remote --strict
```

脚本负责检查：

- 表和字段是否存在。
- `verb_paths` 与 `verbs` 是否正确关联。
- `steps_json`、`growth_json` 是否为合法 JSON。
- 节点、连线和步骤是否完整且引用有效。
- 动作节点数量、主干连线和修饰连线方向是否正确。
- 每一步展示顺序、重点节点和新增内容是否有效。
- 主干句、完整句是否与第一步和最后一步一致。
- 兼容 `steps_json` 是否与 `growth_json.steps` 保持一致。
- 基础空白、标点、重复句子和禁用语法术语问题。

脚本输出分为：

- `error`：确定性结构错误，必须修复。
- `warning`：需要人工确认的潜在文本问题。

自动校验通过不代表语言质量已经通过。英文搭配、场景自然度、翻译准确度和语义关系仍必须按照第 11 节进行人工复核。

## 16. 批量生成脚本

项目提供：

```text
scripts/generate-verb-paths.mjs
```

推荐使用两个并发生成任务，每批处理 20 个动词：

```bash
npm run verbs:generate -- \
  --batch-size=20 \
  --concurrency=2 \
  --attempts=4 \
  --provider=codex \
  --model=gpt-5.5 \
  --reasoning=high
```

模型生成会并发执行，本地严格校验和远端 D1 写入会自动串行执行，避免 SQLite 写锁和远端导入冲突。每个成功批次会立即写入远端，任务中断后重新运行同一命令即可从缺失动词处继续。

常用参数：

- `--batch-size`：每批动词数量，默认 `20`。
- `--concurrency`：并发生成批次数量，默认 `2`。
- `--attempts`：单批生成和修复的最大尝试次数，默认 `4`。
- `--limit`：仅处理前 N 个缺失动词，适合试跑。
- `--dry-run`：生成并完成本地严格校验，但不写入远端 D1。
- `--local-only`：跳过远端 D1 写入。
