# 句子结构 SVG 绘制规范

> 版本：1.0  
> 状态：正式采用  
> 适用范围：学习策略页、动词详情页、例句详情页以及后续所有句子结构 SVG

本文件是句子结构分析与 SVG 生成的唯一权威规范。

历史规则中与本文件冲突的内容全部废弃，包括但不限于：

- 主干关系可以省略箭头。
- 每个句子只能包含一个动作节点。
- 为适应较小画布而合并、隐藏或弱化重要关系。
- 使用固定模板快速生成相似树状结构。

后续新增、重做和人工校验的句子结构图，均必须以本文件为准。

## 1. 目的

本规范用于统一 English Orbit 的句子结构图。

结构图不是把句子拆成几个方块，也不是追求视觉对称，而是要准确回答：

1. 句子里有哪些动作？
2. 每个动作带出了哪些核心部分？
3. 每个修饰内容真正说明的是哪个节点？
4. 句子中是否存在嵌套动作、共享参与者或完整的时间事件？

最终目标是让学习者不依赖复杂语法术语，也能沿着箭头看懂句子的真实关系。

## 2. 核心原则

### 2.1 先分析关系，再绘图

禁止先套固定模板，再把单词填入节点。

每个句子必须先完成以下分析：

1. 找出所有具有独立事件意义的动作。
2. 确定主要动作和嵌套动作。
3. 确定每个动作直接带出的核心部分。
4. 确定每个修饰内容真正说明的节点。
5. 识别共享参与者、时间事件和容易产生歧义的归属关系。
6. 最后才确定节点坐标和 SVG 曲线。

### 2.2 语义准确优先于版面对称

- 不为了左右对称改变真实关系。
- 不把所有绿色节点机械连接到最上方动词。
- 不因为某条线较长，就把箭头改指向更近但错误的节点。
- 必要时允许结构图向下增长，以换取清晰关系。

### 2.3 一张图表达完整关系

每个案例使用一张独立 SVG。

图中应同时包含：

- 完整英文句子
- 所有动作节点
- 动作带出的核心节点
- 重要修饰节点
- 明确的场景标签
- 正确的箭头方向

不再把同一个案例拆散到多个无关联模块中。

## 3. 视觉语言

| 类型 | 推荐样式 | 含义 |
|---|---|---|
| 动作节点 | 红色椭圆 | 句子中的动作或状态变化，如 `runs`、`send`、`changed` |
| 核心节点 | 蓝色圆角矩形 | 动作直接带出的参与者或内容 |
| 修饰节点 | 绿色圆角矩形 | 频率、范围、位置、时间、所属等补充信息 |
| 核心关系 | 黑色实线箭头 | 从动作指向它直接带出的核心节点 |
| 修饰关系 | 绿色实线箭头 | 从修饰内容指向真正被修饰的节点 |
| 共享关系 | 深灰曲线，可使用虚线 | 同一个节点同时参与另一个动作 |

颜色只用于区分关系，不承担唯一信息。节点上必须保留文字和场景标签。

## 4. 箭头方向

### 4.1 动作带出核心部分

核心关系使用：

```text
动作 → 动作带出的核心部分
```

例如：

```text
runs → team
runs → test suite
send → the report
send → the client
```

黑色箭头表达的是“这个动作涉及谁或什么”，不是“谁修饰谁”。

### 4.2 修饰内容说明某个节点

修饰关系使用：

```text
修饰内容 → 被修饰节点
```

例如：

```text
usually → runs
full → test suite
Our → team
last night → changed
```

绿色箭头的终点必须落在真正被说明的节点上。

### 4.3 完整事件修饰另一个事件

从句或事件块不能被压缩成一个模糊的“时间”节点。

先把内部事件画完整，再让整个事件指向它说明的动作：

```text
changed → we
changed → the configuration
last night → changed

整个 changed 事件 → stopped working
```

这表示：

- `last night` 说明什么时候改变配置。
- `after we changed the configuration last night` 整体说明服务器什么时候停止工作。

### 4.4 共享参与者

当一个节点同时参与两个动作时，应显式连接两次，不依赖学习者自行推断。

例如：

```text
asked → me
send → me
```

第二条关系可标注：

```text
me 也是 send 的执行者
```

## 5. 节点拆分规则

### 5.1 动作必须独立成节点

以下内容应独立成动作节点：

- `runs`
- `asked`
- `send`
- `show`
- `stopped working`
- `changed`

动词短语表达一个完整动作时可以保留为一个节点，例如：

```text
stopped working
```

### 5.2 核心参与者保持稳定词块

核心节点应当是容易整体理解的词块：

```text
team
test suite
the report
the client
the logs
the server
the configuration
```

### 5.3 修饰内容尽量拆到真正目标

例如：

```text
Our → team
full → test suite
project → manager
```

不要将 `Our team`、`full test suite`、`project manager` 永远合并成无法继续分析的大节点。

但如果继续拆分不会带来学习价值，可以保留稳定词块，避免过度碎片化。

## 6. 场景标签

每个节点上方应使用简短中文说明其当前作用。

推荐标签包括：

- 动作核心
- 嵌套动作
- 时间事件
- 谁做
- 做什么
- 谁发起
- 请谁
- 发送什么
- 发给谁
- 什么显示
- 显示的事件
- 谁停止
- 谁改变
- 改变什么
- 频率
- 范围
- 所属
- 领域
- 位置 / 环境
- 时间
- 什么时候之前

标签应贴合当前句子变化，禁止所有句子机械使用同一套标签。

## 7. 三个基准案例

这三个案例是后续 SVG 生成与验收的基准。

### 7.1 测试流程

完整句：

```text
Our team usually runs the full test suite on the release branch before each release.
```

关系：

```text
runs → team
runs → test suite

Our → team
full → test suite
usually → runs
on the release branch → runs
before each release → runs
```

注意：

- `usually` 说明运行频率。
- `on the release branch` 说明运行测试时所处的位置或环境。
- `before each release` 说明运行测试的时间。
- 三者都指向 `runs`，不能指向整个句子卡片。

### 7.2 任务传递

完整句：

```text
The project manager asked me to send the report to the client before Friday’s meeting.
```

关系：

```text
asked → manager
asked → me
asked → send

project → manager

send → me
send → the report
send → the client
before Friday’s meeting → send
```

注意：

- `send` 是独立的嵌套动作，不能只作为一段普通绿色修饰文本。
- `me` 是 `asked` 涉及的人，也是 `send` 的实际执行者。
- `before Friday’s meeting` 更自然地说明发送报告的截止时间，因此指向 `send`。

### 7.3 日志与状态变化

完整句：

```text
The logs show that the server stopped working after we changed the configuration last night.
```

关系：

```text
show → the logs
show → stopped working

stopped working → the server

changed → we
changed → the configuration
last night → changed

整个 changed 事件 → stopped working
```

注意：

- `show`、`stopped working`、`changed` 都是动作节点。
- `last night` 优先指向 `changed`。
- `after we changed the configuration last night` 整体说明 `stopped working` 的时间。
- 不能把整段内容粗略合并成一个“什么时候停止”节点后结束分析。

## 8. SVG 绘制要求

### 8.1 布局

- 主要动作位于上方或视觉中心。
- 嵌套动作形成独立的下一层结构。
- 核心节点与动作保持合理距离。
- 修饰节点放在被修饰节点外围。
- 页面允许向下增长，不强行压缩到单屏高度。
- 节点、文字和曲线不得重叠。

### 8.2 曲线

- 使用贝塞尔曲线，不使用僵硬的直角折线。
- 曲线需要有轻微人工绘制感，但不能抖动到影响阅读。
- 同一层曲线避免完全平行和机械对称。
- 曲线起点和终点必须清晰落在对应节点边缘。
- 箭头不得停在空白区域，也不得被其他节点遮挡。

### 8.3 动画

- 可以使用 `stroke-dasharray` 与 `stroke-dashoffset` 模拟线条绘制。
- 节点出现顺序应跟随关系分析，而不是固定模板。
- 动画只用于帮助观察结构，不能改变关系或箭头方向。
- 必须支持 `prefers-reduced-motion`。

### 8.4 可访问性

- SVG 必须包含 `<title>`。
- `role="img"`。
- `aria-labelledby` 指向标题。
- DOM 中的节点文本和关系说明应可被辅助技术读取。
- 颜色之外必须有形状、箭头和文字共同表达含义。

## 9. 统一展示规则

学习策略页、动词详情页、例句详情页、人工校验页面以及后续新增页面，统一使用同一套关系表达：

- 动作使用红色椭圆。
- 核心关系使用黑色实线箭头。
- 修饰关系使用绿色实线箭头。
- 每条箭头都必须有明确的语义起点和终点。
- 主要动作、嵌套动作、共享参与者和修饰归属均应完整展示。

逐步播放只控制节点与关系的出现顺序，不得改变最终结构：

1. 第一步可以只展示主要动作。
2. 后续步骤逐渐加入核心参与者、嵌套动作和修饰内容。
3. 已出现的核心关系必须按照“动作 → 核心部分”展示黑色箭头。
4. 已出现的修饰关系必须按照“修饰内容 → 被修饰节点”展示绿色箭头。
5. 最后一步必须还原完整、严谨的关系树。

页面空间不足时，应优先采用以下方式：

- 增加 SVG 高度。
- 调整节点层级与间距。
- 允许页面纵向滚动。
- 将较长文本合理换行。

不得通过省略箭头、合并独立动作、隐藏重要节点或改变关系方向来压缩画面。

## 10. 生成流程

每次生成句子 SVG，必须按以下顺序执行：

1. 写出完整句子。
2. 列出全部动作。
3. 确定主要动作。
4. 确定每个动作带出的核心部分。
5. 列出每个修饰内容。
6. 为每个修饰内容写出唯一目标节点。
7. 检查嵌套动作和共享参与者。
8. 先用文本形式列出全部关系。
9. 再设置节点坐标。
10. 绘制核心箭头。
11. 绘制修饰箭头。
12. 检查每个箭头终点。
13. 在桌面浏览器中完成视觉检查。
14. 运行 `lint` 和 `build`。

## 11. 验收清单

### 关系

- [ ] 所有动作均已识别。
- [ ] 主要动作清楚。
- [ ] 嵌套动作独立成节点。
- [ ] 核心箭头从动作指向参与者或内容。
- [ ] 修饰箭头从修饰内容指向真正目标。
- [ ] 共享参与者被显式表达。
- [ ] 时间短语没有挂错动作。
- [ ] 从句内部关系没有被粗略合并。

### 视觉

- [ ] 箭头方向一眼可辨。
- [ ] 箭头终点接触正确节点。
- [ ] 曲线没有穿过无关节点。
- [ ] 线条和文字没有重叠。
- [ ] 图形不依赖对称才能成立。
- [ ] 结构过长时允许页面向下扩展。
- [ ] 动作、核心部分和修饰内容的颜色与形状一致。

### 内容

- [ ] 节点文本与原句一致。
- [ ] 没有遗漏影响理解的核心内容。
- [ ] 没有为了画面简洁改变句意。
- [ ] 场景标签简短、准确且不机械。
- [ ] 学习者不需要掌握复杂语法术语。

## 12. 禁止事项

- 禁止所有修饰节点都连接到主动作。
- 禁止箭头方向根据版面方便随意决定。
- 禁止只画“主干 + 三个分类框”而不表达内部关系。
- 禁止将多个独立动作合并成普通文本节点。
- 禁止使用固定模板批量生成所有句子。
- 禁止为了对称改变节点归属。
- 禁止在未经关系分析的情况下直接生成 SVG。
